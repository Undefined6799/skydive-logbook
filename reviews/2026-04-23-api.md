## OpenAPI spec snapshot

**Generated spec inspection (static analysis):**

- **openapi version:** Claimed 3.1 in CLAUDE.md ¬ß1 and openapi.py ¬ß99. 
  - *Code check:* `get_openapi()` is FastAPI's standard function (defaults to 3.1 on FastAPI ‚â•0.100). The spec is not hand-edited to downgrade. **Result: 3.1.x is served.**
  
- **API version:** `0.1.0` (openapi.py:16, rest.py:29)

- **Paths and operations:**
  - 6 operations across 4 path patterns (health, list jumps, create/read/update/delete).
  - All jump operations have `summary` (required for OpenAPI hygiene).
  - All jump operations have implicit `operationId` derived from handler function name (FastAPI auto-assigns).
  - Health endpoint lacks tags but is marked `meta`.

- **Error schemas:**
  - `ProblemDetails` component defined and hand-authored (openapi.py:34-96). Matches RFC 9457 shape.
  - Shared response references (NotFound, Conflict, ValidationFailed, IntegrityError) defined in components.responses (openapi.py:129-143).
  - **Problem:** Jump routes do not explicitly declare their error responses in the OpenAPI signature. Routes like `POST /api/v1/jumps` can return 409 (jump_number_conflict per D23) but the spec has no `responses` metadata. FastAPI infers only success (201) from `status_code` and `response_model`. Error responses are handled at runtime by `on_service_error` and never appear in the spec.

- **Request schemas:**
  - JumpCreate, JumpUpdate, JumpSummary, Jump are Pydantic models (imported from backend.models.jump). FastAPI auto-generates schemas from these.
  - No anonymous inline types detected.

- **Runtime errors generating the spec:**
  - None observed from the code. Spec generation is deterministic from route signatures.

**Verdict: 3.1 spec is served; error schemas are modeled; but error responses are not declared in route signatures, making them invisible to OpenAPI consumers.**

## RFC 9457 conformance findings

### Requirement checklist

Per RFC 9457 ¬ß3:
- ‚úì Media type is `application/problem+json` (errors.py:45, rest.py:64-68).
- ‚úì Required fields `type`, `title`, `status` always present (errors.py:62-84, build_problem()).
- ‚úì Optional `detail` present on every error (exc.message ‚Üí detail).
- ‚úì Optional `instance` present (request path, rest.py:67).
- ‚úì Extensions `code` (stable), `request_id` (UUID), `errors` (array of field errors) documented.

### Per-route error paths

**All routes:**
- Route handlers do not emit HTTPException or raw JSONResponse. All errors are `ServiceError` subclasses caught by `on_service_error` (rest.py:45-68).
- Response is always `error_response(exc, ...)` which yields RFC 9457 body with correct media type.

**POST /api/v1/jumps (create_jump_route, jumps.py:131-171):**
- Can raise `ValidationFailedError` (multipart jump field parse, jumps.py:89-110).
- Can raise `JumpNumberConflict` (service layer, D23).
- Can raise `IntegrityError` (XML validation, disk write).
- Can raise `NotFoundError` (attachment reference not found ‚Äî deferred in v0.1).
- **Status:** All map to problem+json via `on_service_error`. No bare exceptions leak.
- **Spec gap:** OpenAPI declares `response_model=Jump` and `status_code=201` but not the 409, 422, 500 responses that ServiceError subclasses emit. A client reading the spec would not know these are possible.

**GET /api/v1/jumps (list_jumps_route, jumps.py:185-193):**
- Query params `limit` and `offset` are Pydantic-validated by FastAPI (automatic 422 if out of range).
- Service call `jump_service.list_jumps()` does not raise (returns empty list on no matches).
- **Status:** No errors path. Spec is complete.

**GET /api/v1/jumps/{jump_id} (get_jump_route, jumps.py:208-213):**
- `jump_id` is a UUID (automatic 422 if malformed).
- Service call can raise `NotFoundError` (jump not found) or `IntegrityError` (XML parse failure).
- **Status:** Errors map to problem+json. **Spec gap:** 404 and 500 not declared.

**PUT /api/v1/jumps/{jump_id} (update_jump_route, jumps.py:232-238):**
- Body is `JumpUpdate` (automatic 422 if invalid).
- Service can raise `NotFoundError`, `JumpNumberConflict`, `IntegrityError`.
- **Status:** **Spec gap:** 404, 409, 500 not declared.

**DELETE /api/v1/jumps/{jump_id} (delete_jump_route, jumps.py:254-262):**
- Service can raise `NotFoundError`, `IntegrityError`.
- **Status:** **Spec gap:** 404, 500 not declared.

### Summary

**Violations:** None. All errors conform to RFC 9457 at runtime.

**Spec incompleteness:** Successful responses (201, 200, 204) are declared; error responses (4xx, 5xx) are not. The OpenAPI schema does not enumerate possible error codes. A third-party client reading `/openapi.json` cannot predict which error codes a route may emit. **This is a documentation bug, not a conformance bug.**

**v0.1 requirements (CLAUDE.md ¬ß10):**

2. Upload files to a jump

## Summary

**OpenAPI version:** 3.1.x (confirmed served)

**RFC 9457 violations:** 0 (all errors conform at runtime)

**RFC 9457 spec gaps:** 6 routes do not declare error responses in OpenAPI signatures (medium severity; runtime behavior is correct, spec is incomplete)

**D7 thinness violations:** 0 (all routes are thin: parse, call service, return)

**D14 v0.1 scope gaps:** Equipment, stats, file endpoints are v0.1 scope but not yet wired (intentional per Phase comments)

**Biggest gap:** OpenAPI error-response schema is defined but not attached to routes. Third-party clients cannot discover error codes from the published spec.
